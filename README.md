# Web QA Automation Testbed

## 기획 의도 및 배경
"반복되는 수동 테스트, 어떻게 하면 효율적으로 자동화할 수 있을까?"

개발 과정에서 회귀 테스트(Regression Test)를 사람이 매번 수동으로 확인하는 것은 시간 소모가 크고 휴먼 에러가 발생할 가능성이 높습니다. 이러한 문제를 해결하고자, '실제 이커머스와 유사한 테스트베드를 만들고 이를 검증하는 E2E 자동화 파이프라인을 도입하자'는 핵심 아이디어를 기획했습니다.

이 프로젝트는 '문제 정의와 비즈니스 요구사항(What/Why)은 사람이, 세부적인 기술적 의사결정과 구현(How)은 AI가' 담당하는 협업 방식을 잘 보여주는 포트폴리오입니다. 저는 테스트해야 할 핵심 기능과 풀고자 하는 문제를 제시(Prompting)했고, 상세한 시스템 아키텍처나 구조 설계 등은 AI의 기술적 추천을 적극 수용하여 완성했습니다.

---

## 프로젝트 설계 및 협업 방식
저는 프로덕트 오너(PO)처럼 기획과 요구사항을 정의하고, AI를 테크 리드(Tech Lead) 겸 개발자로 활용하여 프로젝트를 진행했습니다.

1. 테스트베드 구축: 제가 "이커머스의 핵심 여정이 포함된 쇼핑몰을 만들어달라"고 요구사항을 정의하자, AI가 최신 웹 트렌드를 반영하여 React 19와 Vite 기반의 아키텍처를 추천하고 전체 코드를 구현했습니다.
2. 테스트 도구 및 환경 세팅: "안정적이고 빠른 E2E 테스트 환경이 필요하다"는 저의 목표를 바탕으로, AI가 여러 도구(Cypress, Selenium, Playwright)의 장단점을 분석해 주었습니다. 그중 비동기 처리에 가장 유리한 Playwright를 AI의 추천에 따라 채택하고 세팅을 위임했습니다.

## 주요 테스트 시나리오 기획
실제 쇼핑몰 이용자의 핵심 흐름을 검증하기 위해 제가 큰 틀에서 정의한 6가지 비즈니스 요구사항(User Journey)입니다. (이에 대한 상세 구조 설계와 실제 테스트 스크립트 작성은 모두 AI가 수행했습니다.)

- 인증 (auth.spec.ts): 회원가입, 로그인, 로그아웃 기능 정상 작동 요구
- 상품 목록 및 탐색 (products.spec.ts): 상품 목록 노출, 필터링, 검색 기능 요구
- 상품 상세 (product_detail.spec.ts): 개별 상품 정보 확인 및 옵션 선택 요구
- 장바구니 (cart.spec.ts): 장바구니 담기, 수량 변경, 삭제 요구
- 결제 (checkout.spec.ts): 배송 정보 입력 및 결제 완료까지의 비즈니스 흐름 요구
- 엣지 케이스 (edge_cases.spec.ts): 비정상적인 입력 및 네트워크 지연 시 방어 로직 요구

## CI/CD 자동화 (GitHub Actions)
"코드가 푸시될 때마다 자동으로 품질을 테스트해달라"는 저의 기획 방향을 바탕으로, AI가 최적의 CI/CD 파이프라인 구조를 제안하고 구축했습니다.

- 자동 실행 트리거: main 브랜치 푸시 및 PR 생성 시 AI가 작성한 워크플로우에 따라 E2E 테스트가 구동됩니다.
- 테스트 리포트 확인: 테스트 결과(Playwright HTML 리포트)가 GitHub Actions의 Artifacts에 자동으로 업로드되도록 구성하여, 유지보수가 용이한 환경을 마련했습니다.

## 실행 방법

### 1. 패키지 설치
```bash
npm install
```

### 2. 로컬 테스트베드(쇼핑몰 웹) 실행
```bash
npm run dev
```

### 3. E2E 테스트 실행
새 터미널을 열고 아래 명령어를 입력하여 자동화 테스트를 실행해 볼 수 있습니다.

```bash
# 전체 E2E 자동화 테스트 백그라운드 실행
npm run test

# Playwright UI 모드로 테스트 실행
npm run test:ui
```
